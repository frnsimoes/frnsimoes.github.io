108k forks, 134k waits. That’s odd.

fork creates a child process, then returns pid to parent and 0 to child. wait4 blocks the parent until the child exits, collects the exit status, and lets the kernel clean up the process table entry (without it, zombie). The normal flow is 1:1, each fork gets one wait.

More waits than forks means children are dying before the parent calls wait. Parent forks, child dies (or gets killed), wait returns immediately with status.

This is interesting: the delta (134k - 108k = 26k) matches the error column on wait4 - likely ECHILD, “no child processes” because the child already died.

That’s what it was: a script looping and killing processes with >90% CPU plus known malware signatures.
